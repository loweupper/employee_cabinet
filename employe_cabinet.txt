EMPLOYEE CABINET - PROJECT GUIDE
Generated: 2026-03-16

==================================================
1) PROJECT PURPOSE
==================================================
Employee Cabinet is a FastAPI-based corporate web system for:
- authentication and session management
- user profile management
- object and document management
- admin operations (users, permissions, audit logs)
- security monitoring, alerts, and metrics

Core stack:
- Backend: FastAPI, SQLAlchemy, Pydantic v2
- DB: PostgreSQL
- Cache/queues: Redis, Celery
- Templates/UI: Jinja2, Tailwind CSS
- Logging/Monitoring: structured JSON logs, DB audit logs, Prometheus metrics


==================================================
2) HOW THE APP STARTS
==================================================
Entry point:
- app/main.py

Run modes found in project:
- Docker: docker-compose.yml (backend + redis)
- Gunicorn/Uvicorn worker inside container
- Local (dev): python app/main.py (uvicorn.run in __main__)

Key startup behavior in app/main.py:
- initializes logging config (stdout JSON + DB handler + file handlers)
- sets middleware chain (CORS, CSRF, AccessLog, SwaggerSecurity, RequestID)
- mounts static: /static -> app/static
- includes routers:
  - /api/v1 + auth router (/auth/...)
  - /profile + profile router
  - /objects + objects router
  - /documents + documents router
  - admin router (/admin/...)
  - monitoring router only if MONITORING_ENABLED: /admin/monitoring/...
- provides /health and /dashboard endpoints
- starts monitoring init and periodic session cleanup task

7) LOGGING SYSTEM: WHERE IT IS AND HOW TO USE
==================================================
7.1 Where logging is configured
- app/main.py
  - LOGGING_CONFIG dictConfig (stdout json, access handler, database handler)
  - logger names used in project: app, security, system, audit
  - setup_log_handlers(...) adds rotating file handlers in /app/logs/{app,security,system}

7.2 Main logging building blocks
- app/core/logging/actions.py
  - unified API: await log_event(event=..., actor=..., request=..., level=..., create_alert=...)
  - auto-categorizes events (security/admin/system/user)
  - enriches payload with timestamps, request_id, IP, user_agent
  - can auto-create monitoring alerts in DB
- app/core/logging/middleware.py
  - logs every HTTP request + duration + status
  - attempts to bind user context from token
  - writes X-Request-ID response header
- app/core/request_id_middleware.py
  - generates request id and puts it into contextvar + request.state
- app/core/db_log_handler.py
  - writes logs into admin audit table
  - computes TTL by level and sets expires_at
- app/core/logging/handlers.py
  - rotating compressed file handlers
- app/core/logging/filters.py
  - sensitive data redaction and PII masking helpers

7.3 Log storage destinations
- stdout (JSON)
- database (AuditLog)
- files:
  - /app/logs/app/app.log
  - /app/logs/security/security.log
  - /app/logs/system/system.log

7.4 How to use logging in module code
Preferred way for business/security events:
1) import from core.logging.actions:
   from core.logging.actions import log_event
2) in route/service:
   await log_event(
       event="some_event_name",
       actor=current_user_or_none,
       request=request,
       level="INFO",          # or WARNING/ERROR
       create_alert=False,     # True for security alerts
       extra={"key": "value"}
   )

For technical low-level logs also possible:
- logger = logging.getLogger("app")
- logger.info({...}) / logger.warning({...}) / logger.error({...})

Recommendation:
- keep event names consistent (snake_case)
- pass request whenever available (for request_id/IP/user-agent linkage)
- do not put raw secrets/passwords/tokens into logs
